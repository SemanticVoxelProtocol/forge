# SVP Benchmark: 20-Round E-Commerce SaaS

## 目的

量化 SVP 在多轮迭代开发中的架构稳定性优势。

## 设计

### 核心思路

```
准备阶段（人类）：
1. 写完整 OpenSpec（涵盖 20 轮需求）
2. 定义固定 API 端点契约
3. 预写所有 E2E 测试（纯数据驱动）

执行阶段（agent）：
Round N: 拿到需求子集 N → 实现/修改 → 人类跑 E2E tests（累积）
```

### 对比组

| 组 | 描述 |
|---|---|
| **SVP 组** | 使用 SVP 从 L5→L1 单向编译，每轮从上层需求开始 |
| **AI 组** | 纯 AI coding（Claude/GPT + Cursor 等），每轮直接给需求让它改代码 |

### 公平性保障

- 两组看到**同一份 spec**
- 两组必须暴露**同一组 API 端点**
- 测试对两组**完全相同**
- Agent **永远看不到测试**

## 目录结构

```
benchmark/
├── README.md                    ← 你在这里
├── spec/
│   └── openspec.md              ← 完整的 OpenSpec（全 20 轮）
├── contracts/
│   └── api.yaml                 ← API 端点契约（OpenAPI 3.1）
├── rounds/
│   ├── round-01.md              ← 每轮给 agent 的需求文档
│   ├── round-02.md
│   └── ...
└── tests/
    ├── data/
    │   ├── round-01.json        ← 每轮的测试数据（input → expected）
    │   ├── round-02.json
    │   └── ...
    └── runner/
        └── run.ts               ← 通用测试 runner
```

## 量化指标

| 指标 | 计算方式 |
|---|---|
| 通过率 | `当轮通过数 / 当轮总测试数` |
| 累积回归率 | `旧测试失败数 / 旧测试总数` |
| 稳定性曲线 | 20 轮通过率折线图 |
| 最终得分 | Round 20 的全量测试通过率 |

## 20 轮需求概览

### Phase 1: 地基（R1-R5）

| 轮次 | 需求 | 考察点 |
|---|---|---|
| R1 | 租户注册 + 用户认证（JWT） | 基础 auth |
| R2 | 商品目录 CRUD | 基础实体 |
| R3 | 购物车 | 一对多关联 |
| R4 | 下单（购物车 → 订单） | 跨实体操作、事务 |
| R5 | 支付（mock 网关） | 外部集成 |

### Phase 2: 业务规则（R6-R10）

| 轮次 | 需求 | 考察点 |
|---|---|---|
| R6 | 库存管理 | 事务一致性 |
| R7 | 角色权限（admin/seller/buyer） | RBAC |
| R8 | 订单状态机 | 状态流转 |
| R9 | 搜索 + 过滤 + 分页 | 复杂查询 |
| R10 | 优惠券系统 | 跨实体业务规则 |

### Phase 3: 跨切面（R11-R15）

| 轮次 | 需求 | 考察点 |
|---|---|---|
| R11 | 多租户数据隔离 | 全局架构变更 |
| R12 | 审计日志 | 跨切面中间件 |
| R13 | 通知系统 | 事件驱动 |
| R14 | 批量操作 | 批处理 |
| R15 | 商品变体（SKU） | Schema 迁移 |

### Phase 4: 架构压力（R16-R20）

| 轮次 | 需求 | 考察点 |
|---|---|---|
| R16 | 评价系统 | 新旧实体交叉 |
| R17 | 卖家数据看板 | 聚合查询 |
| R18 | 退款流程 | 逆向操作耦合 |
| R19 | 多币种支持 | 全局 retrofit |
| R20 | API v2 + v1 兼容 | 版本管理 |
