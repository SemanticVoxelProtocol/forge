# SVP 完整思路

> 这篇文档完整描述 SVP 的设计思路和核心理念。面向想要理解"SVP 在做什么、为什么这么做"的读者。不涉及具体实现细节。

---

## 一、问题

AI 会写代码了。而且写得越来越好。

但有一个问题始终没解决：**人类怎么在 AI 写代码的过程中保持控制？**

不是说 AI 不够好——恰恰相反，AI 越强，这个问题越严重。当 AI 一次性生成几千行代码，人类面对的不再是"这行写得对不对"，而是"这坨东西整体上符合我的意图吗"。AI 犯的错不再是语法错误，而是**理解偏差**——它以为你要的是 A，其实你要的是 B。

传统的代码审查手段在这里失效了。你不可能逐行审查 AI 生成的几千行代码。你需要一种方式，在不同的抽象层级上观测 AI 的工作，快速定位"它在哪一步跑偏了"。

这就是 SVP 要解决的问题。

---

## 二、核心理念：透明优先于正确

SVP 不试图让 AI 不犯错。这是不现实的，也不是正确的目标。

SVP 的目标是：**当 AI 犯了错，人类能立即看到、定位到、修复掉。**

这意味着 SVP 不是一个"让 AI 更准"的工具，而是一个"让错误无处隐藏"的框架。AI 可以犯任何错误，但这个错误必须是可见的、可定位的、可修复的。

这个理念决定了 SVP 的所有设计选择。

---

## 三、五层数据模型：五个观测窗口

SVP 把软件系统分成五个抽象层级：

```
L5  Blueprint    系统要做什么——意图、边界、领域拓扑
L4  Logic Chain  流程怎么编排——步骤、顺序、数据传递
L3  Logic Block  每个单元做什么——输入输出、校验规则、约束条件、逻辑描述
L2  Code Block   代码怎么组织——文件映射、对账哈希
L1  Code         最终的源代码
```

这五层不是防线，不是为了"层层把关"。它们是**五个观测窗口**。

当系统出了问题，你不用面对一坨代码去猜，而是逐层问：

- **意图对吗？**（L5）——系统应该做什么，有没有理解错
- **流程对吗？**（L4）——步骤的编排、顺序、并行/串行是否合理
- **契约够吗？**（L3）——每个单元的输入输出、校验规则、约束条件是否完整
- **结构对吗？**（L2）——代码的组织方式、文件划分是否合理
- **实现忠实吗？**（L1）——代码是否忠实地实现了上层的契约

问到哪层答案变了，问题就在哪层。这比"在几千行代码里找 bug"高效得多。

### 本质是三层

五层展开来看是五个观测窗口，但本质上是三个层次：

```
意图层   L5 + L4    做什么、怎么编排     人定义
契约层   L3         每个单元的规格       人定义，AI 验证
实现层   L2 + L1    代码结构和源码       AI 生成，人验收
```

L3 是整个体系的**枢纽**。它上面是人的意图（做什么），下面是 AI 的实现（怎么做）。L3 契约盒就是人和 AI 之间的接口——人在 L3 写清楚"我要什么"，AI 在 L3 以下实现"怎么做"。

五层而不是三层，是因为每个层次内部还需要不同粒度的观测：
- 意图层拆成 L5（系统级意图）和 L4（流程级编排），是因为"系统要做什么"和"流程怎么走"是两个不同的观测维度
- 实现层拆成 L2（文件映射）和 L1（源代码），是因为"代码怎么组织"和"代码写了什么"需要分开追踪变更

### 层间关系

五层之间不是简单的"每层聚合下一层"：

```
L5 ─聚合─→ L4 ─聚合─→ L3 ←1:1配对→ L2 ─聚合─→ L1
```

- L5 包含多个 L4 flow（一个系统有多个流程）
- L4 包含多个 L3 block（一个流程有多个步骤）
- L3 和 L2 是 1:1 配对（一个逻辑单元对应一个代码实现）
- L2 包含多个 L1 文件（一个代码实现可能有多个源文件）

L3 和 L2 的关系值得单独说。它们不是上下级，而是**同一个单元的两面**：L3 说"做什么"（契约），L2 说"怎么做"（实现映射）。分开是因为它们的编辑者不同（L3 由人定义，L2 由 AI 生成）、变更频率不同（L3 改得少，L2 改得多），分开后可以独立追踪变更。

### 契约盒模型（L3）

L3 是整个体系的核心层。每个 L3 block 是一个**契约盒**——它精确定义了一个单元应该做什么，但不说怎么做。

```
┌─────────────────────────────────┐
│  契约盒                          │
│                                 │
│  pins:        输入输出的数据形状   │  ← 结构化，机器可验证
│  validate:    输入约束            │  ← 自然语言规则
│  constraints: 输出约束 + 不变量   │  ← 自然语言断言
│                                 │
│  description: 中间逻辑            │  ← 自然语言，AI 编译
└─────────────────────────────────┘
```

**validate 约束输入，constraints 约束输出，description 负责中间。**

pins 是结构化的（类型、名称），可以被工具自动校验。validate、constraints、description 是自然语言——不是因为偷懒，而是因为 AI 直接理解自然语言，结构化反而丢失语义（后面会展开）。

当 AI 根据契约盒生成代码时，可能实现得不完美。但 validate 和 constraints 构成的契约可以自动验证——这才是形式化的真正价值。**不是"帮 AI 理解"，而是"AI 理解错了的时候能被发现"。**

## 四、AI 即编译器

这是 SVP 最核心的设计哲学。

传统编译器是确定性的——同样的输入永远产生同样的输出。所以它需要精确的格式：严格的语法、形式化的类型系统、无歧义的规范。

AI 编译器是理解性的——它读懂意图，然后实现。所以它需要的是**清晰**，不是**严格**。

### 为什么不用 JSON Schema / Protocol Buffers / 自定义 DSL

传统做法会选择 JSON Schema（形式化校验）、Protocol Buffers（跨语言序列化）或自定义 DSL（领域专用语法）来定义数据模型。SVP 都不用，因为这些工具是为传统 parser 设计的，AI 不需要。

SVP 的数据载体是 TypeScript 对象。但 TypeScript 在这里的角色不是"实现语言"——它是**对 AI 最高效的类型描述载体**。AI 在训练数据中见过数十亿个 TypeScript interface，看到 `interface OrderRequest { user_id: string; items: LineItem[] }` 就自然知道怎么转成任何语言。

### 为什么 validate 和 constraints 用自然语言

对比：

```
// SVP 选择的（AI 直接理解）：
validate: { "request.items": "array, min 1, max 50" }

// 传统做法（为 parser 设计）：
validate: [{ field: "request.items", rule: "range", params: { type: "array", min: 1, max: 50 } }]
```

两种写法包含完全相同的信息。但第一种人和 AI 都能直觉理解，第二种只是在给不存在的 parser 喂食。

更重要的是 constraints：

```
constraints: ["output.result.errors contains all failed checks, not just first"]
```

"not just first"这个微妙但重要的语义暗示，在结构化表示中会丢失。AI 完全理解这句话的含义，结构化反而是降级。

### 结构化的边界

"AI 不需要结构化"不意味着什么都不结构化。分界标准很简单：**层间连接关系必须结构化，block 内部可以是黑盒。**

需要结构化的（工具要校验的）：
- pins 的类型——连线两端类型是否兼容
- wire 的引用——from/to 指向的 pin 是否存在
- 层间引用——L4 step 引用的 L3 block 是否存在

不需要结构化的（展示给人看就行）：
- validate 的规则
- constraints 的断言
- description 的逻辑

工具对黑盒部分只做一件事：**展示**。不解析、不校验、不转换。

## 五、定位：增强层，不是 AI 管线

SVP 不自己调 AI API。SVP 不造编译器。

SVP 是 AI 编码工具（Claude Code、Cursor 等）的**增强层**。类似 [OpenSpec](https://github.com/Fission-AI/OpenSpec) 的定位——不造 AI，给 AI 喂更好的上下文。

SVP 的工具链分两面：面向 AI 的和面向人的。

### 面向 AI 的工具

SVP 的数据模型本身就是为 AI 优化的——TypeScript 类型定义、自然语言契约、结构化的层间引用。AI 编码工具直接消费这些数据。

- **`svp view`**（虚拟文件树）：把 `.svp/` 下的 JSON 数据实时渲染成 AI 友好的聚焦视图。AI 在某一层看到该层全貌，按需切换层级。
- **`svp compile-plan`**（编译计划）：检测变更范围，输出结构化的任务清单——哪些实体需要重编译、每个任务需要的最小上下文、任务间的依赖关系。AI 编码工具根据计划派发 subagent。
- **`svp check`**（校验）：编译后的自动验收——hash 一致性、引用完整性、漂移检测、图结构。AI 跑 check 就知道自己做对了没有。
- **store**（数据读写）：读写 `.svp/` 下的五层数据。
- **hash**（变更追踪）：计算和比对 contentHash。

### 面向人的工具：查看器

SVP 的数据是为 AI 对齐的（TypeScript 对象、JSON 持久化）。人看 JSON 不直观。所以 SVP 提供**查看器**——把 AI 对齐的数据渲染成人类对齐的视图。

查看器**只读不写**。编辑是 AI 的事（通过 `svp view` + `svp edit` 操作数据）。查看器只是让人能看懂 AI 在操作什么。

- **svp-blueprint**（蓝图查看器）：L4/L3 的可视化节点图。把 flow 的步骤编排和 block 之间的数据流渲染成节点和连线，让人一眼看到全局拓扑。人在这个视图里理解系统结构，但不直接在图上编辑——要改的话，在对应层级告诉 AI 去改。
- **CLI 查看器**：`svp status` 展示各层的健康状态（哪些 hash 匹配、哪些有 drift）。文本界面，适合快速检查。
- **未来可以有更多查看器**：Web dashboard、IDE 插件、移动端概览。底下读的是同一份数据，只是渲染方式不同。

查看器的价值是**让人能观测**。SVP 的核心理念是透明——你得看得见才能判断对不对。查看器就是五个观测窗口的具体实现。

### 定位的含义

这个定位意味着：
- SVP 的能力随 base model 进化自动提升——模型越强，理解契约越准，生成代码越好
- SVP 不锁定任何 AI 提供商——今天用 Claude Code，明天用别的，SVP 数据模型不变
- SVP 是语言无关的协议——TypeScript 只是第一个实现工具链的语言

## 六、交互架构：逐层渗透

这是 SVP 与 AI 协作的核心模型。

### 聚焦视图

SVP 不让 AI 自由浏览五层数据。SVP 给 AI **聚焦的视野**——在某一层看到该层的全貌，需要时才切换层级。

为什么？研究表明 LLM 不擅长处理太多上下文。更少但更精准的上下文反而准确率更高。SVP 在每一层只展示该层的信息，AI 集中注意力在一个抽象层级上工作。

`svp view` 命令提供两级视图：
- **Overview**：一个实体一行——名字、签名、摘要统计。AI 看到全局拓扑。
- **Detail**：展示本层完整信息，但不展开其他层。用导航标记指向关联层，AI 按需切换。

视图格式针对 AI 优化——像代码不像 JSON，因为 AI 训练数据中代码和 Markdown 最多。

### 三条规则

1. **用户在哪层，AI 从下一层进入**
2. **严格单向向下，不回头**
3. **做不到就报错，说清哪层有什么问题**

### 逐层渗透

当用户在某层做了修改，SVP 计算变更范围，AI 从下一层开始逐层向下编译：

```
用户在 L5 → AI 入口 L4 → L3 → L2 → L1    （全量编译）
用户在 L4 → AI 入口 L3 → L2 → L1          （流程级编译）
用户在 L3 → AI 入口 L2 → L1               （单 block 编译）
用户在 L2 → AI 入口 L1                    （纯代码生成）
用户在 L1 → AI 不介入，用户自己改代码       （手动模式）
```

**关键设计：每层用独立的 subagent 处理。**

主 Agent 只待在用户所在层，上下文极少（只有该层的拓扑信息）。每层的编译由独立的 subagent 完成，每个 subagent 只看到自己那层的最小必要信息。

这带来几个好处：

**上下文效率**：最重的上下文（L1 源代码，可能几百行）离用户最远，隔离在 subagent 里。用户手边的上下文（L4 拓扑，可能 20 行）极其精简。用户作为指挥官，永远不会被代码细节淹没。

**天然并行**：多个 block 的编译可以并行——3 个 L3 block 改了，就同时跑 3 个 subagent。

**失败隔离**：某个 L1 subagent 生成的代码有问题？只需重跑那一个。契约（L3）没变，重编译成本很低。

**强制结构化思考**：AI 不能"偷懒"直接改 L1 代码然后声称完成了。改需求必须先改 L4 流程，必须更新 L3 契约，然后才改代码。每层的变更都有记录。

### 为什么严格单向

如果 L1 subagent 发现 L3 的契约有问题（比如缺一个输入参数），它**不修改 L3**，而是报错返回。

原因很简单：上层是人编辑的意图。L3 的契约是用户定义的规格，AI 不应该未经许可修改。subagent 报错 → 用户看到 → 用户在对应层修 → 重新向下编译。

**用户就是反向反馈回路。** SVP 不需要造一个自动的反向传播机制。这比自动修复更安全，也更符合"人类不失控"的目标。

### 编译计划

用户修改某层后，`svp compile-plan` 计算需要重编译的范围，输出结构化的任务清单。每个任务标注操作类型、原因、需要的最小上下文、依赖关系。AI 编码工具根据这个计划派发 subagent，无依赖的任务并行执行。

越高层介入，编译深度越深，消耗越大。越低层介入，越精确，越便宜。用户自然会学会在合适的层级操作。

## 七、变更追踪

每层的数据模型包含 `contentHash`——内容的哈希值。变更追踪基于哈希比对。

**自上而下（重编译）**：上层的 contentHash 变了 → 下层记录的 sourceHash 不匹配 → 触发重编译。这是 AI 自动执行的。

**自下而上（对账检测）**：下层被手改 → 下层的 contentHash 和上层记录的不匹配 → 标记 drift 警告。这只是检测机制，不自动修复。用户看到 drift 后自己决定怎么处理。

`svp check` 是编译后的验收工具，校验四类问题：

1. **Hash 一致性**——contentHash 和实际内容是否匹配
2. **引用完整性**——层间引用的实体是否存在，pin 类型是否兼容
3. **漂移检测**——上下层的哈希是否同步
4. **图结构**——流程是否有环、是否有孤立步骤

check 报什么层有问题，用户就去那层看。这是逐层渗透模型的闭环：

```
用户改上层 → AI 向下编译 → svp check 校验 → 有问题回到对应层修 → 再次向下编译
```

## 八、存储原则：存得少，算得多

很多看起来应该存储的信息，SVP 选择在用到时计算：

- L3 的函数签名 → 从 input/output 实时生成
- L4 聚合了哪些 L3 blocks → 从 steps 的引用收集
- L5 包含哪些 L4 flows → 从 L4 的来源引用反算
- DataFlow 的数据类型 → 从引用的 pin 类型算出来

**单一数据源 + 计算，永远不会有不一致的风险。** 存两份数据就有两份数据不一致的可能。只存一份，其他的都算出来。

## 九、设计原则总结

1. **透明优先于正确** — 不追求 AI 不犯错，追求错误可见可定位
2. **每层都是观测窗口** — 五层结构的价值是分层观测，不是分层防御
3. **AI 即编译器** — 格式设计优化 AI 理解，不是 parser 解析
4. **只做真正的声明式** — 伪声明式不如自然语言
5. **存得少算得多** — 单一数据源 + 计算，避免不一致
6. **协议与实现分离** — SVP 是语言无关的规范，工具链是独立的
7. **用户是反馈回路** — 严格单向编译，AI 不越权修改上层意图

---

## 十、SVP 不做什么

明确 SVP 的边界，和它容易被误解的地方：

- **不调 AI API** — SVP 只提供工具链和结构化上下文，不自己跑 AI
- **不造编辑器** — svp-blueprint（可视化编辑器）是可选的编辑视图，不是 SVP 核心
- **不定义新语言** — 数据载体是 TypeScript 对象，不发明 DSL
- **不做运行时** — SVP 是开发时工具，不参与代码的运行
- **不锁定提供商** — 今天用 Claude Code，明天用别的，SVP 数据不变
- **不追求零错误** — 追求的是错误透明，不是错误消灭

---

## 十一、一句话总结

SVP 是五层观测框架。它让人类在 AI 写代码时，能在不同抽象层级上看到 AI 在做什么、做对了什么、做错了什么，然后精确地介入修复。AI 越强，SVP 越有用——因为 AI 生成的代码越多，分层观测的价值就越大。
