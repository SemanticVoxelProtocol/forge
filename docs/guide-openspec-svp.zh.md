# 搭配使用 OpenSpec 与 SVP

OpenSpec 和 SVP 解决的是同一个问题的两半。

**OpenSpec** 帮你在实现开始前把功能应该做什么说清楚。
**SVP** 帮你在实现之后继续保证系统结构和实现保持一致。

搭配使用时，它们形成一条实用流程：明确行为 → 设计结构 → 实现 → 验证。

## 各自解决什么问题

### OpenSpec —— 需求与行为

当你需要先把功能行为讲清楚，再进入实现时，优先使用 OpenSpec。

它最擅长：

- 用可审查的方式描述预期行为
- 记录功能意图和验收场景
- 让实现工作围绕已确认的需求展开

比如你说“加个深色模式”，OpenSpec 会先帮你把这个需求具体化成可讨论、可确认的行为。

### SVP —— 结构与一致性

当你需要保证实现结果持续符合系统结构时，优先使用 SVP。

它最擅长：

- 从高层理解系统结构
- 保持模块边界和职责清晰
- 检查实现是否偏离原本的结构设计

当 AI 写完代码后，SVP 能帮助你更早发现架构漂移或模块边界被破坏的问题。

### 两者如何配合

| 关注点 | OpenSpec | SVP |
|---|---|---|
| 定义功能行为 | 主工具 | 提供补充上下文 |
| 明确业务预期 | 主工具 | 不是重点 |
| 理解系统结构 | 提供补充上下文 | 主工具 |
| 保持实现不跑偏 | 提供补充上下文 | 主工具 |
| 同时验证行为与结构 | 可配合 SVP | 可配合 OpenSpec |

**一句话：** OpenSpec 关注“软件应该做什么”，SVP 关注“实现如何保持结构清晰、持续可控”。

## 推荐工作流

### 1. 先用 OpenSpec 定义功能

当团队需要先把功能行为讲清楚时，从 OpenSpec 开始。

例如要给结账流程加入优惠券功能，可以先在 OpenSpec 中明确：

- 什么情况下允许使用优惠券
- 什么情况算有效或无效优惠券
- 校验成功或失败时用户会看到什么结果

这样实现阶段就有了稳定的行为目标。

### 2. 再用 SVP 约束实现结构

当行为已经清楚后，再用 SVP 把实现放进系统结构里。

同样以优惠券功能为例，SVP 更适合帮助你思考：

- 优惠券校验应该放在哪个流程位置
- 哪个模块应该负责这件事
- 这个模块应该暴露哪些输入和输出

这样可以避免功能做出来了，但结构越来越难维护。

### 3. 实现后同时验证

当行为和结构都明确后：

- 完成功能实现
- 用 OpenSpec 确认结果仍符合预期行为
- 用 SVP 确认实现仍符合预期结构

两者一起使用，可以同时降低“做错功能”和“做对功能但结构变乱”的风险。

## 初始化设置

### 新项目同时使用两者

```bash
# 初始化 OpenSpec
openspec init --tools claude

# 初始化 SVP
forge init --name my-project --host claude-code
```

初始化后：

- `openspec/` 负责需求与行为管理
- `.svp/` 负责结构与一致性验证

### 已使用 OpenSpec 的项目加入 SVP

```bash
forge init --name my-project --host claude-code
```

如果项目已经有现成代码，先用 SVP 帮你建立稳定的结构视图，再逐步依赖它做一致性检查。

### 已使用 SVP 的项目加入 OpenSpec

```bash
openspec init --tools claude
```

这样可以补上需求层，而不改变 SVP 原本承担的结构验证角色。

## 什么时候用哪个

| 你想…… | 用 |
|---|---|
| 定义一个功能应该做什么 | OpenSpec |
| 在写代码前明确验收行为 | OpenSpec |
| 理解功能应该怎样放进系统结构 | SVP |
| 保持实现不偏离系统结构 | SVP |
| 验证产品行为是否符合预期 | OpenSpec |
| 验证架构和实现是否一致 | SVP |
| 同时管理需求和结构 | OpenSpec + SVP |

## 总结

```
OpenSpec          SVP
────────          ───
应该做什么        如何保持结构清晰
写代码之前        实现过程中与实现之后
行为清晰          结构一致

        搭配使用
        ────────
        行为 → 结构 → 实现 → 验证
```
